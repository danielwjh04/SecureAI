# SP3.5 — LLM Policy Authoring — Design Spec

**Status:** Approved (2026-06-26).
**Depends on:** SP1 (audit chain), SP2 (policy IR + enforcer), SP3 (model provider).
**Delivers:** natural-language operator intent → a strictly-validated, tool-grounded `PolicySchema` proposal, with a two-step human-gated propose→activate flow and tamper-evident activation provenance.

---

## 1. Decisions locked (from brainstorming)

| Question | Decision |
|---|---|
| How does the local model produce structured policy reliably? | **Grammar-constrained generation** (GBNF/JSON-schema grammar) + a `generate(prompt, *, grammar)` method added to the existing `ModelProvider` seam, so authoring is swappable too. Parsed output is validated against `PolicySchema` as a second gate. |
| Human-in-the-loop activation | **Two-step propose → activate.** The engine returns a validated `PolicyProposal` + diff (testable core); a thin CLI stages it (`policies/proposed/`) and a separate explicit `activate` promotes it into `policies/`. Nothing auto-loads. |
| Input contract & field scope | **Grounded.** A tool inventory (names + descriptions) is required; the grammar constrains tool-name fields to that inventory and a post-parse gate rejects any tool not in it. Authors `denylist`, `external_comms_tools`, `tool_rules`, `taint_sources`, `content_scan_sources`. **Not** `injection_signatures` (curated separately). |

**Why a weak local model is still safe here:** the model only *proposes*; a human *disposes*. The grammar guarantees structural validity, two validation gates reject malformed or hallucinated output, and nothing activates without an explicit human step. The architecture is the asset; the local model is a swappable stand-in.

## 2. Module layout

```
secureSG/
  models/
    provider.py     # +abstract generate(prompt, *, grammar) -> str
    guardformer.py  # +generate via create_completion(grammar=...); pure prompt build tested,
                    #   native call exercised with a fake _CompletionModel (as for assess)
  guard/
    policy.py       # PolicySchema, CompiledPolicy, load_policy (EXTRACTED from enforcer.py)
    enforcer.py     # imports the IR from policy.py (behavior-preserving)
  warden/
    __init__.py
    grammar.py      # build a tool-grounded grammar + JSON schema from the inventory
    authoring.py    # PolicyAuthor pipeline + PolicyProposal / PolicyDiff
schemas/ (unchanged)
scripts/
  policy_author.py  # CLI: `propose "<intent>"` and `activate <file>`
policies/
  proposed/         # staging dir; load_policy globs "*.yaml" non-recursively, so it never auto-loads
```

**Policy-IR extraction (refactor):** `warden/authoring` is the second consumer of `PolicySchema`/`load_policy` (the enforcer is the first), so the IR moves to its own `guard/policy.py`. `enforcer.py` imports it; every SP2 test stays green. Same justified-by-a-second-consumer extraction as the SP3 Aho-Corasick move. No behavior change.

## 3. Provider extension (`models/provider.py`, `guardformer.py`)

`ModelProvider` gains:
```python
@abstractmethod
async def generate(self, prompt: str, *, grammar: str | None = None) -> str: ...
```
`QwenGuardProvider.generate` calls `create_completion(prompt, grammar=LlamaGrammar.from_string(grammar) if grammar else None, temperature=0.0, max_tokens=settings.model_author_max_tokens)` and returns the completion text. Prompt construction is a pure, unit-tested function; the native call is exercised through a fake `_CompletionModel` (no weights), mirroring `assess`. The test `StubProvider` gains a scripted `generate`.

> Library-syntax note: the exact grammar API (`LlamaGrammar.from_string` / `from_json_schema`, and whether key-constraints use JSON-schema `propertyNames` or hand-built GBNF) is confirmed against the installed llama-cpp-python at implementation. No method signature is invented.

## 4. The authoring pipeline (`warden/authoring.py`) — defense in depth

`async author_policy(provider, intent, *, tools, current_policy) -> PolicyProposal`:

1. **Grammar** from the inventory (`grammar.py`): tool-name fields constrained to the tool enum; values constrained to the verdict enum (`ALLOW`/`BLOCK`/`HUMAN_APPROVAL_REQUIRED`) and tier enum (`LOW`/`MEDIUM`/`HIGH`).
2. **Generate** grammar-constrained JSON via `provider.generate`.
3. **Parse** — `json.loads`; malformed ⇒ `AuthoringError` (fail-closed).
4. **Structural gate** — `PolicySchema.model_validate` (types + `extra="forbid"`).
5. **Grounding gate** — every tool referenced (in any of the five fields) must be in the inventory, else `AuthoringError`. Belt-and-suspenders beyond the grammar.
6. **Diff** vs `current_policy` → `PolicyDiff`.
7. Return frozen `PolicyProposal(policy, diff, intent)`.

The proposal is a **delta fragment** that merges via `load_policy` (the existing merge model), not a full replacement.

### Data contracts
- `PolicyProposal` (frozen): `policy: PolicySchema`, `diff: PolicyDiff`, `intent: str`.
- `PolicyDiff` (frozen): per-field additions and changes vs the current merged policy (lists: new entries; dicts: new keys and `old→new` value changes), with a human-readable `render()`.

## 5. Staging & activation (`scripts/policy_author.py`)

- `propose "<intent>"` — runs the pipeline against the active policy + a configured tool inventory, writes the validated proposal to `policies/proposed/<slug>.yaml` with the **intent and provenance as a header comment**, and prints the diff. Nothing is active.
- `activate <file>` — re-loads and re-validates the staged file (it must still parse as policy), promotes it into `policies/`, and appends a **tamper-evident activation record to the SP1 audit chain** (`AuditRecord`, `verdict=ALLOW` as a neutral marker, `details = {event: "policy.activated", intent, policy_sha256}`). "Who changed the rules, from what intent, when" becomes provable — the Verifiable-Enforcement thesis applied to governance.

---

## 6. Settings / exceptions

- Settings: `model_author_max_tokens: int = 512`, `proposed_policy_dir: Path = policies/proposed`.
- New exception `AuthoringError(SecureSGError)` for any generation/parse/validation/grounding failure. The pipeline never yields a partial or unvalidated policy.

---

## 7. Error handling / fail-closed

- Malformed JSON, schema-invalid output, or any reference to a tool outside the inventory ⇒ `AuthoringError`; no proposal is produced.
- A staged file that no longer parses ⇒ `activate` refuses and raises.
- Activation auditing shares the SP1 chain's atomic, idempotent append.

---

## 8. Testing

`StubProvider.generate` returns scripted JSON — no weights. Coverage:
- `grammar.py`: the built grammar/schema contains the tool enum, verdict enum, and tier enum; an empty inventory is handled.
- `authoring.py`: happy path → valid `PolicyProposal`; malformed JSON → `AuthoringError`; schema-invalid → `AuthoringError`; **hallucinated/ungrounded tool → `AuthoringError`**; diff correctness (additions + value changes).
- `guardformer.generate`: prompt build (pure) + generation via a fake `_CompletionModel`.
- CLI: `propose` writes a staged file with provenance and does not touch `policies/`; `activate` promotes it, audits to the chain, and the chain still verifies `CHAIN_OK`.
- One `@pytest.mark.model`-gated real-model authoring test (skipped without weights).
- Gates: coverage ≥ 85%, ruff, `mypy --strict`.

---

## 9. Scope boundary (explicit deferrals)

- No `injection_signatures` authoring (curated threat list, not NL-authored).
- No Warden discovery / intent-drift / embeddings (SP4); the tool inventory is supplied to `propose`, not auto-discovered yet.
- No dashboard authoring UI (SP6); the CLI is the human gate this cycle.

**Future work (not this cycle):** *adversarial policy testing* — after authoring, have the model generate example tool-calls the proposed policy would block/flag, so a reviewer sees the policy's real effect alongside the diff. Strengthens the human gate; deferred to keep this cycle focused.

5. **`warden/authoring.py`** — `PolicyProposal`, `PolicyDiff`, `author_policy` pipeline with both validation gates (RED→GREEN).

## 11. Success criteria

- All SP1–SP3 tests stay green; every new public function has a unit test.
- A plain-language intent over a tool inventory yields a schema-valid, tool-grounded proposal; malformed or hallucinated output is rejected with `AuthoringError`.
- `propose` never activates; `activate` promotes + audits, and the chain verifies.
- Coverage ≥ 85%, ruff clean, `mypy --strict` clean.
